## 2. Why 12,390 missing cell files are not actionable

The Dru_map reference has 4130 lotheader, 4130 lotpack, and 4130 chunkdata files.
These represent Dru_map's proprietary built-world binary data. They cannot be
copied into PZMapForge for the following reasons:
- They are third-party mod assets (Dru_map by its author).
- They are specific to Drummondville, not PZMapForge's content.
- Copying them would constitute distributing someone else's mod.

The PZMapForge candidate already has its own coordinate-aligned binary files
(`35_27.lotheader`, `world_35_27.lotpack`, `chunkdata_35_27.bin`) from the
MAP-7U staging pass. These are the correct binary files to test.

---

## 3. Sidecar stubs probe rationale

The remaining non-cell sidecar gap (8 files) is the next testable variable.
These stubs are generated from scratch — they are not copied from Dru_map.

Hypothesis: PZ may require `streets.xml.bin`, `worldmap.xml.bin`, or
`worldmap-forest.xml.bin` to be present in the map folder for IsoMetaGrid
to recognize and mount the map folder.

---

## 4. Generated sidecar stubs

All stubs are deterministic and generated from scratch by the staging script:

| File | Type | Content |
|---|---|---|
| `streets.xml.bin` | minimal binary stub | ASCII marker: PZMF_MAP7Y_STUB_streets_xml_bin |
| `worldmap.xml.bin` | minimal binary stub | ASCII marker: PZMF_MAP7Y_STUB_worldmap_xml_bin |
| `worldmap-forest.xml.bin` | minimal binary stub | ASCII marker: PZMF_MAP7Y_STUB_worldmap_forest_xml_bin |
| `worldmap.png` | generated placeholder PNG | 64×64 dark-gray placeholder |

**These are NOT copied from Dru_map.** They are candidate-owned diagnostic
stubs generated by PZMapForge tooling under `.local/` only.

The `.bak` variants (`worldmap.xml.bak`, `worldmap-forest.xml.bak`, etc.) are
not included in this first probe. They are likely authoring pipeline artifacts,
not runtime registration inputs.

## 6. Test outcomes

### Outcome 1: Success (map mounts or binary evidence appears)

```text
Candidate map folder appears in IsoMetaGrid scan
OR: expected_map_lotheader_meta_evidence_found=true
OR: explicit candidate lotheader parse error appears
OR: explicit sidecar parse/read error appears
```

If sidecar parse error appears → sidecar presence is read/parsed but format wrong.
If lotheader parse error appears → binary writer gate opens.
If map folder mounts → sidecar stubs were the discriminator.

### Outcome 2: Partial (sidecar read error)

PZ tries to read/parse a sidecar stub but fails due to invalid format.
Evidence: file-read error referencing `streets.xml.bin` or `worldmap.xml.bin`.
Next: investigate the actual binary format of these files.

### Outcome 3: No change

Workshop Ready + mod loaded + spawn honored + fallback forest + empty scan.
Same result as K004/K006. Sidecar presence alone was not the discriminator.
Next: server-side log capture, alternative registration contract investigation.
