# Phase 3: Local PZ Install Config Spec

Date: 2026-06-01
Status: PRECONDITION DOCUMENTATION ONLY -- not implemented
Author: operator

---

## Claim boundary

planning_artifact_only_not_pz_load_tested

All work in Phase 3 produces local planning artifacts only.
No work in Phase 3 produces a playable Project Zomboid export until a real
local load test is performed and documented as a separate evidence record.

---

## Why this exists

Phase 2A and 2B established a stable, well-tested multi-layer semantic planning
pipeline. The pipeline outputs planning artifacts that use generated colour-strip
tiles for TileZed visualization. These tiles are correct for planning review but
are not Project Zomboid game tiles.

Phase 3 begins preparing a bridge from semantic kind -> planning tile toward
semantic kind -> local PZ tile GID. This bridge requires knowledge of the
operator's local PZ installation without copying, redistributing, or committing
any PZ game assets.

This document specifies the contract for a local-only configuration file that
enables Phase 3 work while preserving all existing safety boundaries.

---

## What Phase 3 is allowed to do

- Read a local operator-provided config pointing to a PZ install directory.
- Discover tilesheet files (*.pack, *.tiles, or equivalent) in the local install.
- Build a local-only tile reference catalog (artifact in .local/, not committed).
- Map semantic planning kinds to tile references found in the local install.
- Produce a planning artifact that records which local tile IDs correspond to
  which semantic kinds. This artifact is .local/ only and is not committed.
- Produce a planning TMX that references local tile GIDs for TileZed review.
- Report tile reference gaps where no local tile maps to a semantic kind.

---

## What Phase 3 is not allowed to do

- Copy, commit, or redistribute any PZ tilesheet, texture, or game asset.
- Write to media/maps or any game installation directory.
- Generate lotpack, lotheader, or bin files.
- Claim compatibility with a PZ build version without a documented local load test.
- Claim the tool is official Project Zomboid tooling.
- State or imply that a generated artifact is playable without a real load test.
- Hard-code PZ install paths or asset checksums in committed source files.
- Embed any PZ asset content in the repo, even as test fixtures.
- Use network calls to locate or download PZ assets.

---

## Proposed local config file path

    .local/pzmapforge/pz-install-config.json

This file is:
- Written by the operator, not generated by any tool.
- Gitignored (.local/ is already ignored).
- Never committed.
- Loaded at runtime only when Phase 3 commands are invoked.

If the file is absent, Phase 3 commands print a clear error:
    "pz-install-config.json not found. See docs/PHASE_3_LOCAL_PZ_CONFIG_SPEC.md."

---

## Proposed config schema shape

    {
      "schema": "pzmapforge.local-pz-install-config.v0.1",
      "claim_boundary": "planning_artifact_only_not_pz_load_tested",
      "pz_install_root": "E:/SteamLibrary/steamapps/common/ProjectZomboid",
      "tiles_root": "media/tiles",
      "allow_asset_copy": false,
      "allow_media_maps_write": false,
      "tile_reference_mode": "local_reference_only",
      "notes": "Local operator config. Do not commit."
    }

Field definitions:

  schema
    Must be "pzmapforge.local-pz-install-config.v0.1".

  claim_boundary
    Must be "planning_artifact_only_not_pz_load_tested".

  pz_install_root
    Absolute path to the local Project Zomboid installation directory.
    Not validated for existence at config parse time; validated at use time.
    Must not be a network share or cloud sync path during active discovery.

  tiles_root
    Path relative to pz_install_root where tilesheet files are located.
    Default: "media/tiles".
    The tool reads only, never writes, this directory.

  allow_asset_copy
    Must be false. Copying PZ assets is forbidden.
    A validator must fail if this field is true.

  allow_media_maps_write
    Must be false. Writing to game directories is forbidden.
    A validator must fail if this field is true.

  tile_reference_mode
    "local_reference_only" means tile IDs are referenced for planning artifacts
    only. The tile image data is not read or copied.
    Other values are reserved and must be rejected.

  notes
    Free text. Operator note. Not read by any tool.

## Local PZ install path handling

Rules:
- The path in pz_install_root is resolved to an absolute path at use time.
- The tool checks that pz_install_root exists as a directory before proceeding.
- The tool does not attempt to read or validate the path at config load time.
- The path is local operator state and must never appear in committed artifacts.
- The path must not be included in any report or artifact that could be committed.
  Instead, reports use a placeholder: "[local PZ install path]".

---

## Tilesheet discovery rules

- The tool discovers tilesheet files (*.pack or *.tiles) by listing the tiles_root
  directory under pz_install_root.
- Discovery is read-only. No files are written, moved, renamed, or opened for
  write access during discovery.
- The tool reads only file names and sizes, not file contents, during discovery.
- File contents are read only when building a tile reference catalog and only
  for the purpose of extracting tile GID tables.
- No tilesheet content is stored in the catalog beyond tile names, GIDs, and
  the sheet file name it came from.
- The catalog is written to .local/mapforge/ and is gitignored.

---

## Tile reference rules

- A tile reference is a (tilesheet_name, gid) pair.
- Tile references are stored in the local tile reference catalog, not in
  committed source files.
- A semantic kind maps to zero or more tile references.
- If no tile reference exists for a kind, the kind is marked unmapped in the
  catalog and the planning artifact notes the gap.
- The mapping from kind to tile reference is the operator's responsibility.
  PZMapForge provides the tooling to discover and record the mapping; it does
  not hard-code any PZ tile semantics.

---

## No-copy asset rule

PZMapForge must never copy, embed, cache, or commit any PZ tilesheet, tile
texture, or game asset. This applies to:
- Test fixtures
- Committed schemas
- Report content
- Generated artifacts
- Any intermediate build output

The only permissible interaction with PZ assets is read-only at runtime from
a locally installed PZ copy, and only for the purpose of building a local tile
reference catalog that is itself gitignored.

---

## No media/maps write rule

No PZMapForge command, script, or test may write to any directory named or
located under media/maps, regardless of the local config.

allow_media_maps_write must be false in the config. A validator must fail and
refuse to proceed if this field is missing or set to any other value.

---

## No playable export claim rule

No Phase 3 artifact may be described as playable, load-tested, or compatible
with any specific PZ build version unless:
1. A real local load test has been performed.
2. The load test result is documented in a committed evidence record.
3. A decision record ratifies the new claim boundary.

Until those three conditions are met, all Phase 3 artifacts must carry:
  claim_boundary = "planning_artifact_only_not_pz_load_tested"

This rule cannot be overridden by configuration.

---

## Safety checks before any tile mapping

Before any Phase 3 command may proceed, it must verify:

1. The local config exists at .local/pzmapforge/pz-install-config.json.
2. schema == "pzmapforge.local-pz-install-config.v0.1".
3. claim_boundary == "planning_artifact_only_not_pz_load_tested".
4. allow_asset_copy == false.
5. allow_media_maps_write == false.
6. tile_reference_mode == "local_reference_only".
7. pz_install_root exists as a local directory.
8. tiles_root directory exists under pz_install_root.

If any check fails, the command exits with a clear, specific error message and
returns exit code 1. No tile discovery or mapping proceeds until all checks pass.

---

## Evidence required before implementation

Phase 3 implementation must not begin on any slice until the following are in
place:

1. This spec document is ratified (committed in main, reviewed).
2. A local PZ installation is accessible to the operator.
3. A tilesheet layout survey has been performed and documented.
   (What directory structure does the local install use?
    What file formats are present: .pack, .tiles, other?)
4. A tile naming convention spec exists for the semantic kinds.
   (Which PZ tiles correspond to grass, road, row_house, etc.?)
5. A Phase 3A decision record is written before Slice 3A-1 begins.
   The decision record must document the operator's answers to 2-4.
